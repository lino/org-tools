* Introduction and Goals

** Requirements Overview

org-tools is a unified CLI for org-mode files written in Rust. It provides
linting, formatting, querying, clocking, and exporting — all without
requiring Emacs.

Key requirements:

- *Standalone*: no Emacs dependency — runs as a single binary
- *Fast*: process large org-mode collections in sub-second time
- *Correct*: rule behaviour matches the org-mode specification and Emacs
  org-lint where feasible
- *Safe*: never corrupt user files; protected regions (source blocks, example
  blocks, etc.) are never modified by format rules
- *Configurable*: rules can be enabled/disabled via =.org-tools.toml=; defaults
  match Emacs org-mode behaviour
- *Composable*: commands produce output in formats suitable for piping
  (=org query --format=locator | org clock report --from-locators -=)

** Quality Goals

| Priority | Quality Goal  | Description                                                     |
|----------+---------------+-----------------------------------------------------------------|
|        1 | Correctness   | Rules match org-mode spec and Emacs behaviour                   |
|        2 | Safety        | Never modify content inside protected regions                   |
|        3 | Performance   | Sub-second for typical org-mode collections (hundreds of files) |
|        4 | Extensibility | Adding a new rule requires only a struct + trait impl + tests   |
|        5 | Composability | Commands pipe together via locators and sparse org subtrees     |

org-tools reads =.org= files from the filesystem, applies format/lint/query
operations, and outputs results to stdout. Commands compose via locator strings
and sparse org subtrees on stdin/stdout.

** Technical Context

org-tools is a pure command-line tool. Its only I/O is:

- *Input*: =.org= files from the local filesystem, =.org-tools.toml= configuration,
  stdin (for piped org content or locator lists)
- *Output*: diagnostics, query results, locators, formatted files to stdout/stderr

No network access, no database, no external APIs.

* Solution Strategy

The project is structured as a *Cargo workspace* with a shared library and an
umbrella binary:

- =org-tools-core=: all parsing, rules, document model, locator resolution
- =org=: unified CLI with subcommands (=fmt=, =query=, future: =clock=, =update=, =export=)

** Lint & Format Strategy

Line-based text processing with span-based fixes:

1. Read file into =SourceFile= (indexes line starts for O(log n) lookups)
2. Run *format rules* → collect =Fix= structs (byte span + replacement)
3. Deduplicate and apply fixes in a single pass
4. Run *lint rules* on formatted result → collect =Diagnostic= structs
5. Output in requested format

** Query Strategy

1. Parse all =.org= files into =OrgDocument= heading trees (single-pass scan)
2. Parse query string into a =Predicate= AST (recursive descent)
3. Evaluate predicates against each =OrgEntry= (with tag inheritance)
4. Output matches in requested format (human, JSON, locator, org subtree)

** Level 4: Rule Architecture

Every rule is a struct implementing =FormatRule= or =LintRule=:

#+begin_src rust
pub trait FormatRule: Send + Sync {
    fn id(&self) -> &'static str;          // e.g. "F001"
    fn name(&self) -> &'static str;        // e.g. "trailing-whitespace"
    fn description(&self) -> &'static str;
    fn format(&self, ctx: &FormatContext) -> Vec<Fix>;
}

pub trait LintRule: Send + Sync {
    fn id(&self) -> &'static str;          // e.g. "W001"
    fn name(&self) -> &'static str;        // e.g. "heading-level-gap"
    fn description(&self) -> &'static str;
    fn check(&self, ctx: &LintContext) -> Vec<Diagnostic>;
}
#+end_src

Rules are independent — no rule depends on another rule's output.

* Deployment View

org-tools is distributed as a single statically-linked binary. No runtime
dependencies, no configuration files required (defaults are sensible).

Deployment targets:

- *Local development*: installed via =cargo install= or downloaded binary
- *CI/CD*: run in pipeline steps; use =--format json= for machine parsing
- *Editor integration*: invoked as an external formatter/linter
- *Shell scripts*: pipe locators between commands

* Cross-cutting Concepts

** Protected Regions

Format rules must never modify content inside org-mode blocks where the
content is literal text (source, example, export, quote, verse, center,
comment blocks). The =regions.rs= module scans for =#+BEGIN_= / =#+END_=
pairs and returns line ranges. Format rules call =is_protected()= to skip
these lines.

** Fix Application

Fixes are span-based: each =Fix= specifies a byte range and its replacement.
The formatter applies fixes in a single left-to-right pass, which requires
fixes to be sorted by start position and non-overlapping. The runner handles
deduplication by keeping the first fix when spans overlap.

** Configuration Hierarchy

org-tools searches for =.org-tools.toml= starting from the current directory and
walking up to the filesystem root. The first file found wins. This allows
per-project and per-directory configuration.

** OrgLocator — Universal Entry Addressing

Every heading can be addressed by an OrgLocator string. When an entry has an
=:ID:= property, the locator uses that (survives moves). Otherwise, it falls
back to =CUSTOM_ID=, outline path, or line number. Locators are the glue
between commands: =org query --format=locator= outputs them, and =org clock=
accepts them as input.

** Rule Independence

Rules are fully independent — no rule reads another rule's output. This
makes the system easy to reason about, test, and extend. Each rule is a
self-contained module with its own tests.

* Risks and Technical Debt

| Risk / Debt                      | Mitigation                                        |
|----------------------------------+---------------------------------------------------|
| Line-based parsing has limits    | Sufficient for ~35 of ~58 org-lint checks;        |
|                                  | OrgDocument provides heading-level structure      |
| Table alignment is left-only     | Track as known divergence from Emacs; future fix  |
| No incremental/cached processing | Acceptable at current scale; revisit if needed    |
| No parallel file processing      | Single-threaded is fast enough; easy to add later |
| ID resolution scans all files    | O(n) per lookup; cache if needed at scale         |

* Glossary

| Term             | Definition                                                    |
|------------------+---------------------------------------------------------------|
| Format rule      | A rule that produces =Fix= values to auto-correct formatting  |
| Lint rule        | A rule that produces =Diagnostic= values (report-only)        |
| Protected region | A block (src, example, etc.) whose content must not be        |
|                  | modified by format rules                                      |
| Fix              | A byte-span replacement: replace bytes [start, end) with      |
|                  | new text                                                      |
| Diagnostic       | A reported issue: file, line, column, severity, rule, message |
| Runner           | Orchestrates rule execution and fix application               |
| SourceFile       | In-memory representation of an =.org= file with line index    |
| OrgDocument      | Lightweight heading tree built by single-pass line scanning   |
| OrgEntry         | A heading entry with properties, planning, clocks, tags       |
| OrgLocator       | Universal address for a heading: id, custom-id, path, or line |
| Predicate        | A query condition evaluated against an OrgEntry               |
