#+TITLE: orgfmt — Product Vision
#+DATE: 2026-03-21

* Problem Statement

Org-mode users who want consistent formatting and early error detection
currently rely on Emacs-based tooling (=org-lint=, custom elisp). This creates
friction:

- CI pipelines cannot easily run org-lint without a full Emacs installation
- Non-Emacs editors (VS Code, Neovim, Helix) have no org-mode linting
- Custom formatting scripts (shell, Python, elisp) are fragile and hard to
  maintain

* Vision

orgfmt is the =ruff= / =prettier= of the org-mode world: a fast, standalone
tool that formats and lints org files without requiring Emacs.

* Design Principles

- *Correctness over features*: every rule must match the org-mode
  specification. A rule that produces false positives is worse than no rule.
- *Safety first*: never modify content inside protected regions. Never corrupt
  a file.
- *Sensible defaults*: defaults match Emacs org-mode behaviour. Opinionated
  formatting is opt-in.
- *One rule, one job*: each rule is independent, testable, and documented.
- *Spec-driven*: every rule references the org-mode manual and formal syntax.

* Non-Goals

- Replacing Emacs for editing org files
- Implementing a full org-mode parser
- Supporting org-mode extensions that deviate from the specification
  (though common extensions like org-roam are tracked in the ecosystem doc
  to avoid false positives)
